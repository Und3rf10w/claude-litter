# Event Sourcing for state.json — Design Spike (W7 prep)

## §1 Context

W6 landed `scripts/state-transition.sh` (539 LOC) as the single canonical writer of `state.json`, backed by `_write_state_atomic` flock+tmp+mv from `instance-lib.sh`, with a `state_integrity_hash` tripwire and `frontmatter-gate.sh` blocking direct Write|Edit to `*/state.json`. Every hook that previously used raw `jq+tmp+mv` now calls a `state-transition.sh` subcommand. Reference: `plugins/deepwork/references/single-writer-state-design.md`.

W6 closed the "anyone can freely mutate state.json" vulnerability. It did NOT close a subtler problem: **state drift across partial failures**. If `state-transition.sh` crashes mid-session (power loss, OOM, rogue signal), or if an operator manually patches `state.archived.json` during a post-mortem, the current state.json is an opaque snapshot — there is no audit trail, no way to distinguish "how did we get here" from "this state was written correctly". Reconciliation today means reading log.md markers, which are prose and unreliable.

W7 extends the single-writer invariant to **full event sourcing**: every `state-transition.sh` subcommand invocation appends a structured event to `${INSTANCE_DIR}/events.jsonl` before (or atomically with) writing `state.json`. `state.json` becomes a **projection** — a cached view rebuilt by replaying the event log. The event log is the source of truth. State drift converts from "quietly accumulates" to "structurally impossible — events ARE the truth."

This spike mirrors the structure of `single-writer-state-design.md` (W5.5) and serves the same purpose: de-risk the implementation before W7 code is written.

**Survey baselines (run 2026-04-25):**
- `state-transition.sh` LOC: 539
- JSONL-shaped files already in use: `test-results.jsonl`, `discoveries.jsonl`, `incidents.jsonl`, `metrics-violations.jsonl`, `hook-timing.jsonl`, `change_log.jsonl`, `rollback_log.jsonl` — all written via `>>` (O_APPEND). Pattern is established.
- All 4 hooks that previously called `_write_state_atomic` directly have been migrated to `state-transition.sh` in W6 (`deliver-gate.sh:74`, `execute/plan-drift-detector.sh:61`, `pre-compact.sh:36–38`, `execute/test-capture.sh:149`).
- Zero `event|append|jsonl` hits inside `state-transition.sh` itself — event emission is entirely new work for W7.

## §2 Event Taxonomy

Every `state-transition.sh` subcommand maps to one or more event types. Survey of subcommands and their mutation shapes:

### 2.1 Shared event envelope

All events share these top-level fields:

```json
{
  "event_id":        "<UUID v4 or monotonic counter>",
  "event_type":      "<snake_case type string>",
  "prev_event_hash": "<SHA256 of previous raw line including newline, or GENESIS>",
  "timestamp":       "<ISO-8601 UTC, e.g. 2026-04-25T12:00:00Z>",
  "actor":           "<orchestrator name, hook script basename, or state-transition.sh>",
  "payload":         { /* event-specific fields */ }
}
```

`event_id` is a UUID v4 generated by `state-transition.sh` via `uuidgen` (macOS) or `cat /proc/sys/kernel/random/uuid` (Linux); detect at startup via `command -v uuidgen`. Fallback: `date +%s%N` + PID if neither is available (monotonically unique within a session).

`actor` is set from `${_DW_CALLER:-$(basename "$0")}` — the same convention used by the `set_field` audit log at `state-transition.sh:394`.

## §3 Event Log File Format

### 3.1 File location and naming

```
${INSTANCE_DIR}/events.jsonl
```

One file per deepwork instance. `INSTANCE_DIR` is already set by `discover_instance` for every hook and subcommand invocation.

### 3.2 Encoding

- Append-only JSONL: one event per line, valid compact JSON, terminated by `\n`.
- No blank lines; the hash chain depends on the exact raw bytes of each line including its trailing `\n`.
- Character encoding: UTF-8.

### 3.3 Hash chain

Each event carries `prev_event_hash`:
- First event: `"prev_event_hash": "GENESIS"` (literal string).
- Subsequent events: SHA256 of the raw previous line (bytes as written to disk, including trailing `\n`).

```bash
# Computing prev_event_hash for the next append:
LAST_LINE=$(tail -1 "${INSTANCE_DIR}/events.jsonl" 2>/dev/null || echo "")
if [[ -z "$LAST_LINE" ]]; then
  PREV_HASH="GENESIS"
else
  if command -v sha256sum >/dev/null 2>&1; then
    PREV_HASH=$(printf '%s\n' "$LAST_LINE" | sha256sum | cut -d' ' -f1)
  else
    PREV_HASH=$(printf '%s\n' "$LAST_LINE" | shasum -a 256 | cut -d' ' -f1)
  fi
fi
```

### 3.4 Append atomicity

Shell `>>` on Linux and macOS is O_APPEND, which is atomic for writes smaller than PIPE_BUF (≥512 bytes on all POSIX platforms; 4096+ in practice). Individual events are well under this limit. For events whose payload might exceed PIPE_BUF (e.g., `bootstrap` with a large state snapshot), write to a temp file and `mv` atomically:

```bash
_append_event() {
  local events_file="$1"
  local event_json="$2"
  local byte_count=${#event_json}
  if [[ $byte_count -lt 512 ]]; then
    printf '%s\n' "$event_json" >> "$events_file"
  else
    local tmp="${events_file}.tmp.$$"
    printf '%s\n' "$event_json" > "$tmp" && cat "$tmp" >> "$events_file" && rm -f "$tmp"
  fi
}
```

### 3.5 Why JSONL

- Append atomicity under O_APPEND for short events (POSIX guarantee; already relied on by `test-results.jsonl`, `incidents.jsonl`, etc.).
- Full replay via `jq -s '.'` or `while IFS= read -r line; do ...; done < events.jsonl`.
- Line-addressed — `tail -1` gets the head event; `wc -l` gives event count.
- Human-readable; `jq '.'` formats any line.
- Established pattern in this codebase — no new tooling required.

## §4 Reducer / State Projection

### 4.1 Role

The reducer reads `events.jsonl` in order and produces the current `state.json`. After W7, `state.json` is always a deterministic projection of the event log, not a mutable store.

### 4.2 Implementation: `replay` subcommand in `state-transition.sh`

Rather than a standalone `state-reducer.sh` (reserved for W7 implementation), the reducer is added as a `replay` subcommand to the existing `state-transition.sh`. This keeps the single-writer pattern: the reducer outputs state via the same atomic-write path.

```
state-transition.sh replay [--from-genesis] [--output <path>]
```

- `--from-genesis` (default): replay all events from the start.
- `--output <path>`: write result to `<path>` instead of `$STATE_FILE` (for reconciliation or dry-run).

### 4.3 Reduction rules per event type

The reducer maintains a working state object and applies events in sequence:

| Event type | Reduction rule |
|---|---|
| `bootstrap` | `state = payload.state_snapshot` |
| `phase_advanced` | `state.phase = payload.to_phase` |
| `exec_phase_advanced` | `state.execute.phase = payload.to_phase` |
| `field_set` | `state \| (payload.jq_path) = payload.json_value` |
| `array_appended` | `state \| (payload.jq_path) += [payload.json_object]` |
| `merged` | `state = state * payload.json_fragment` |
| `halt_recorded` | `state.halt_reason = {summary, blockers, recorded_at: timestamp}` |
| `session_backfilled` | `state.session_id = payload.session_id` |
| `flaky_test_added` | `state.execute.flaky_tests += [payload.command]` (if absent) |
| `last_updated_stamped` | `state.last_updated = timestamp` |

`field_set` uses `jq_path` as a jq expression: `jq --argjson val "$json_value" "$jq_path = \$val"`. Since `field_set` paths are logged per `state-transition.sh:396`, the reducer can use the same `jq` path expression directly.

### 4.4 Full-replay vs. incremental reduction

**Full replay**: re-read `events.jsonl` from line 1 on every call to the reducer. Simple; correct by construction. Cost: O(N) where N = event count.

**Incremental**: cache the latest reduction result with the head event hash. On next call, only replay events after the cached head. Cost: O(new events since last reduction).

**Tradeoff**: For the W7 implementation, **full replay is the correct default**. Reasoning:
1. Events per session are bounded in practice (hundreds, not millions). At 1000 events (test ES-g), full replay must complete in <1s — easily achievable with `jq -s`.
2. Incremental reduction requires a valid cache, which complicates failure recovery. If the cache is corrupt or stale, the incremental path silently produces wrong state — worse than the problem we're solving.
3. Full replay gives the reconcile command a clean, unconditional correctness story (§7).

Optimization path: if benchmarking shows >1s replay at realistic session sizes, add periodic snapshot events (§9.4) to bound replay cost, rather than building incremental logic.

### 4.5 `event_head` field in state.json

After every event append + reduction, `state-transition.sh` writes `event_head` to `state.json`:

```json
{
  "event_head": "<SHA256 of the last line of events.jsonl>"
}
```

This is computed as:
```bash
LAST_LINE=$(tail -1 "${INSTANCE_DIR}/events.jsonl")
EVENT_HEAD=$(printf '%s\n' "$LAST_LINE" | sha256sum | cut -d' ' -f1)
```

`state_integrity_hash` (W6) and `event_head` (W7) coexist. `state_integrity_hash` protects the projection-correctness invariant (fields protected: `phase`, `team_name`, etc.). `event_head` is the event-log synchronization check. Hooks verify both.

## §5 Integrity Gate Evolution

### 5.1 Current state (W6)

`frontmatter-gate.sh` (`PreToolUse:Write|Edit`):
- Blocks direct Write|Edit to `*/state.json` unless `_DW_STATE_TRANSITION_WRITER=1`.
- Verifies `state_integrity_hash` matches recomputed value; blocks on mismatch.

`state-transition.sh`:
- Only legal writer of `state.json`.
- Recomputes and writes `state_integrity_hash` atomically on every mutation.

### 5.2 W7 extension

**Option A: extend `frontmatter-gate.sh`** — add `event_head` verification after the existing hash check.

**Option B: new `PreToolUse` hook `event-integrity-gate.sh`** — dedicated hook for event-log checks.

**Recommendation: Option A** — extend `frontmatter-gate.sh`. Rationale:
1. `frontmatter-gate.sh` already fires on `Write|Edit` for `*/state.json` — the exact trigger needed.
2. Adding a second hook doubles subprocess cost per tool call for the same gate condition.
3. The W6 design already established `frontmatter-gate.sh` as the integrity enforcement point. Extending it is lower conceptual surface.

The W7 extension to `frontmatter-gate.sh`:

```bash
# After existing state_integrity_hash check:
EVENT_HEAD_STORED=$(jq -r '.event_head // ""' "$STATE_FILE" 2>/dev/null || echo "")
if [[ -n "$EVENT_HEAD_STORED" ]] && [[ -f "${INSTANCE_DIR}/events.jsonl" ]]; then
  LAST_LINE=$(tail -1 "${INSTANCE_DIR}/events.jsonl" 2>/dev/null || echo "")
  if [[ -n "$LAST_LINE" ]]; then
    if command -v sha256sum >/dev/null 2>&1; then
      ACTUAL_HEAD=$(printf '%s\n' "$LAST_LINE" | sha256sum | cut -d' ' -f1)
    else
      ACTUAL_HEAD=$(printf '%s\n' "$LAST_LINE" | shasum -a 256 | cut -d' ' -f1)
    fi
    if [[ "$EVENT_HEAD_STORED" != "$ACTUAL_HEAD" ]]; then
      printf 'EVENT_HEAD_MISMATCH: state.json event_head does not match events.jsonl tail.\n' >&2
      printf 'Run /deepwork-reconcile to rebuild state.json from the event log.\n' >&2
      exit 2
    fi
  fi
fi
```

Absent `event_head` (pre-W7 instance) is treated as **pass** — same as the absent `state_integrity_hash` convention from W6.

Missing `events.jsonl` with a present `event_head` is a `STATE_DIVERGENCE` condition: the event log was deleted or moved. Gate blocks with the same reconcile prompt.

### 5.3 Gate blocking message

When the gate blocks: `RECONCILIATION_REQUIRED — run /deepwork-reconcile`. This is surfaced in `additionalContext` (exit 2, stderr) so the orchestrator sees it in the tool result.

## §6 Migration Path

### 6.1 Where W6 left migration cost

Post-W6, `state-transition.sh` is the **only writer** of `state.json`. This means the W7 migration is almost entirely **inside `state-transition.sh`**: add event append logic to each subcommand case. No hook call sites need updating — they already call `state-transition.sh`.

Migration cost:
- `state-transition.sh`: add `_append_event` helper + `_compute_prev_hash` helper + event append call in each of the 9 subcommand cases (not `init`). Estimated: +80–100 LOC on top of the existing 539.
- `frontmatter-gate.sh`: ~15 lines for the `event_head` check (§5.2).
- `state.json` schema: add `event_head` field (top-level string). No existing field changes.
- Test fixtures: no change. `init` does not emit events; fixtures continue to write bare JSON via `state-transition.sh init`.

### 6.2 Bootstrap event for in-flight sessions

In-flight sessions at W7 rollout have `state.json` but no `events.jsonl`. Behavior:
1. On the first call to any `state-transition.sh` subcommand post-W7, if `events.jsonl` is absent: generate a `bootstrap` event capturing the full current `state.json` as `payload.state_snapshot`.
2. Write the `bootstrap` event as line 1 of `events.jsonl` (with `prev_event_hash: "GENESIS"`).
3. Proceed with the normal subcommand flow (append the mutation event, reduce, update `state.json`).

```bash
_ensure_event_log() {
  local events_file="${INSTANCE_DIR}/events.jsonl"
  [[ -f "$events_file" ]] && return 0
  # Seed bootstrap event from current state.json
  local state_snap
  state_snap=$(cat "$STATE_FILE" 2>/dev/null) || return 1
  local now event_id boot_event
  now=$(date -u '+%Y-%m-%dT%H:%M:%SZ')
  event_id=$(uuidgen 2>/dev/null || cat /proc/sys/kernel/random/uuid 2>/dev/null || printf '%s-%s' "$(date +%s)" "$$")
  boot_event=$(jq -cn \
    --arg eid "$event_id" \
    --arg ts "$now" \
    --arg actor "${_DW_CALLER:-state-transition.sh}" \
    --argjson snap "$state_snap" \
    '{event_id: $eid, event_type: "bootstrap", prev_event_hash: "GENESIS",
      timestamp: $ts, actor: $actor, payload: {state_snapshot: $snap}}')
  printf '%s\n' "$boot_event" >> "$events_file"
}
```

Call `_ensure_event_log` at the top of each subcommand case (after `_require_state_file`, before `_verify_integrity_hash`).

### 6.3 Backward compatibility

|---|---|
| `events.jsonl` absent + `event_head` absent | Pre-W7 instance; no gate check. First subcommand call seeds the log (§6.2). |

### 6.4 `approve-archive.sh` and event log archiving

`approve-archive.sh:27` renames `state.json` to `state.archived.json` via `mv`. After W7, `events.jsonl` must be archived alongside it:


If `events.jsonl` is absent at archive time (pre-W7 session), the `|| true` silences the error. The archived copy carries the full event history for post-mortem analysis.

## §7 `/deepwork-reconcile` Command

### 7.1 Purpose

Rebuilds `state.json` from `events.jsonl` when the integrity gate blocks or when an operator suspects state drift. This is the only path for recovering from a `STATE_DIVERGENCE` or `EVENT_HEAD_MISMATCH` condition.

### 7.2 Inputs

- Discovers the active instance via `discover_instance` (uses session or team context).
- Reads `${INSTANCE_DIR}/events.jsonl`.
- Accepts `--dry-run` flag to produce a reconciliation report without writing `state.json`.

### 7.3 Behavior

1. Read `events.jsonl` line by line.
2. For each line: parse JSON, verify `prev_event_hash` matches SHA256 of the previous line (or `GENESIS` for line 1).
3. Apply the reduction rule for each event type (§4.3) to the working state object.
4. After all events: write the reduced state to `state.json` via `_write_state_atomic` (with `event_head` and `state_integrity_hash` recomputed).
5. Emit a reconciliation report:
   - Total events processed.
   - Hash chain: `valid` or `BROKEN at event N (event_id=..., expected=..., got=...)`.
   - Divergence: fields that differ between the newly-reduced state and the on-disk `state.json` before reconciliation (diff via `jq`).
   - Final `event_head`.

### 7.4 Implementation path

Implemented as `state-transition.sh replay` (§4.2). The `SKILL.md` wraps this as a user-facing command:

```
plugins/deepwork/skills/deepwork-reconcile/SKILL.md
```

Usage:
```
/deepwork-reconcile [--dry-run]
```

The skill invokes:
```bash
STATE_FILE="$STATE_FILE" bash "${_PLUGIN_ROOT}/scripts/state-transition.sh" replay
```

### 7.5 When to run

- When `frontmatter-gate.sh` emits `EVENT_HEAD_MISMATCH` or `STATE_DIVERGENCE`.
- When hooks log `STATE_DIVERGENCE` warnings to `incidents.jsonl`.
- Manually for debugging — safe to run at any time (idempotent; full replay always produces the same result from the same log).

## §9 Open Questions

The following questions are unresolved and must be decided before W7 implementation begins:

### §9.1 `event_id` format: UUID vs. monotonic counter

**Options:**
- (A) UUID v4 via `uuidgen` — globally unique, no ordering assumption required.
- (B) Monotonic counter (`$(wc -l < events.jsonl) + 1`) — simple, compact, encodes position directly.

**Consideration:** The hash chain already enforces ordering. UUID is globally unique and has no per-session state to manage. Counter requires reading `events.jsonl` before every append (small cost, but adds a read to what should be a write-only append path). `uuidgen` is available on macOS; `/proc/sys/kernel/random/uuid` on Linux. **Recommendation: UUID v4.** The W7 implementer should confirm `uuidgen` availability in the test environment before choosing; if absent on CI, fall back to `openssl rand -hex 16`.

### §9.2 Event append ordering: before or after state.json write?

Two options for the event append + state write sequence:

**(A) Append event → write state.json** — the event is the source of truth; the projection is always a consequence. If state.json write fails after the event appends, the log is ahead of the projection — reconcile replays and fixes. This is the **event-sourcing-correct** order.

**(B) Write state.json → append event** — maintains the W6 guarantee that state.json is always correct before any event is logged. If the event append fails, state.json is correct but the log is behind — reconcile detects `event_head` mismatch and replays.

**Recommendation: Option A** — append event first. The event log is canonical; state.json is the cache. Option B creates a window where state.json is "more advanced" than the log, which contradicts the invariant. Under Option A, a failed state.json write after a successful event append means the next reconcile call rebuilds state.json correctly — no data is lost. This is the standard event-sourcing discipline.

The W7 implementer must decide and document this explicitly in the commit message.

### §9.3 Observer hooks: do they emit events?

PreToolUse and PostToolBatch hooks (`frontmatter-gate.sh`, `batch-gate.sh`, `state-drift-marker.sh`) observe state.json but do not mutate it. Should they emit "observation" events to the log?

**Consideration:** Adding observation events bloats the log with noise that is not useful for replay (reducing them does nothing to state). The events that matter for replay are mutations only. Observer events would only be useful for audit/debugging — that role is already served by `hook-timing.jsonl` and `incidents.jsonl`. **Recommendation: no.** Only `state-transition.sh` subcommands emit events. Observer hooks do not emit to `events.jsonl`.

### §9.4 Snapshot events to bound replay cost

If sessions accumulate thousands of events (long execute phases, high tool-call density), full replay cost grows proportionally. A periodic snapshot event (`state_snapshot`) captures the full projection at a known point; replay can start from the most recent snapshot.

**Consideration:** This adds complexity (the reducer must handle `state_snapshot` events at positions other than line 1). For W7, full replay with the 1000-event performance gate (ES-g) is sufficient. Snapshots should be deferred to W8 if benchmarking shows >1s replay in real sessions. The `bootstrap` event already establishes the snapshot pattern; extending it to periodic snapshots is a straightforward generalization.

**Pre-W7 decision needed:** define the threshold at which a snapshot event is generated (e.g., every 500 events). Even if not implemented in W7, the reducer must be designed to handle `state_snapshot` events at non-genesis positions so the W8 extension is not a breaking change.

### §9.5 Teammate-originated state mutations

Teammates currently call `state-transition.sh` directly via Bash (W6 model). Under full event sourcing, a teammate call to `state-transition.sh phase_advance` emits a `phase_advanced` event with `actor` set to the teammate name (via `_DW_CALLER`).

**Open question:** Should teammates be the direct event emitters, or should they emit "request events" that the orchestrator turns into actual state events? The request-event model adds an extra layer of auditability (you can see "teammate X requested phase advance, orchestrator approved and emitted the event") but doubles the event count for every teammate-originated mutation and requires orchestrator coordination.

**Recommendation for W7:** Keep the direct model — teammates call `state-transition.sh` and emit events with their actor name. The request-event model is a future concern if multi-agent coordination becomes more complex. The `actor` field in the event envelope already distinguishes teammate-originated from orchestrator-originated mutations.

### §9.6 `set_field` and protected fields in event log

`state-transition.sh set_field` currently logs every invocation to `.hook_warnings[]` for audit visibility (§8 Q3 of `single-writer-state-design.md`). After W7, the event log supersedes `hook_warnings[]` as the audit trail for `set_field` calls. The W7 implementer should decide whether to:

**(A)** Remove the `hook_warnings[]` audit append from `set_field` — the event log is the audit trail.
**(B)** Keep both — belt-and-suspenders; `hook_warnings[]` remains browsable in state.json.

**Recommendation: Option B for W7.** Removing the `hook_warnings[]` entry is a behavior change that may break existing log-scraping tools or PROFILE.md recipes that look for `set_field` warnings. Keep both in W7; mark `hook_warnings[]` set_field entries as deprecated in a follow-up.
